# Code structure and assumptions for edit page

Brief explanation on how the edit page works.

## Layers

The edit page implementation is split into horizontal layers, which interact according to specific rules (specified in the next section). 

### Controller

The controller acts simultaneously as a server sync and an independent data store/validator. It has three components:
    - The data manager stores current data, validates operations coming in, and creates requests for the request manager.
    - The request manager schedules/sends requests to the backend and receives responses from the requests generated by the data manager.
    - The ID repository stores client-server id mappings and performs some basic validation with regards to id states.

Only one controller should be created for the edit novel page over its lifecycle (i.e. the time that the end user stays on the edit novel page with a single novel).

Consumers interact with the controller via a pub-sub interface. Consumers pass subscriber functions to the controller, which take a list of getters that return controller state as a parameter along with **trigger events**. Consumers pass messages to the controller via **user events**. From this point forth, we will broadly refer to trigger events as events that bubble outside a given component and user events as events that get passed into a component.

```
                    ┌─────────────────────────────────────┐
                    │            Controller                │
                    │                                     │
  Backend ◄────────►│   ┌──────────┐    ┌──────────────┐  │
```

### Hooks and the Manager

Hooks act as data stores for the UI. These contain react state with which the UI can render directly from, as well as react refs which act as getters for external data consumers.

The manager acts as a bridge between the controller and various hooks. A manager is in charge of providing easy-to-call interfaces to the UI to update state on end user events and take controller event responses to update hooks data. We ensure the following contracts on managers:

The manager is split into a set of submanagers. Submanagers provide slices of the full manager functionality. Specifically, a given submanager is responsible for providing a specific set of related end user event handlers as an interface, along with handling a subset of controller events.

We impose the following restriction on handling controller events using submanagers:

> For a given hook and a given controller trigger event, there can be at most one submanager that updates that hook upon receiving that controller event.

This ensures that on any given controller event, two managers cannot perform operations on a single hook, which is much easier to reason about.

> Important note: the manager (and submanagers) are stateless. State is stored in the hooks.

### Panels

Panels are UI components. They render data using hooks and pass user events to the managers.

### The page itself

The page is (mostly) in charge of initializing everything and arranging the panels. Initialization happens in the following stages:

1. Fetch data from the backend
2. Create the controller using the data
3. Initialize hooks to empty state
4. For each hook, in a `useEffect` (that runs exactly once):
    1. Get relevant data from controller using the controller getters
    2. Set data in the hook according to data retrieved from the controller
5. Create managers
6. Subscribe managers to controller
7. Return page layout and panels using managers for event handlers
